# Task 310: Rename architect-roots personas → pattern-reviewer + drop -roots suffix

## Context

Two existing reviewer personas:

- `agents/board-architect-roots.md` — spec-board specialist for
  structural-pattern-purity at LD-lock time (smell-transfer /
  cycle-symptom-cause / state-vocab-half / etc.). Functionally a
  reviewer.
- `agents/code-architect-roots.md` — code-review-board specialist
  for structural-pattern-purity post-implementation. Same pattern
  classes, different phase. Also functionally a reviewer.

Both names contain "architect" — but the upstream agentic-CLI
convention reserves "architect" for authoring roles (Plan Agent =
"software architect and planning specialist", read-only, AUTHORS
implementation plans). Reviewer roles upstream are named for their
function: `verification-agent` (verifies), worker (works).

The framework's `architect-roots` personas are reviewers, not
authors. The naming is misleading by upstream convention. Adding
`brief-architect` (per spec 306) — a real author role following the
upstream Plan Agent pattern — sharpens the conflict: the new
persona is the only "architect" doing what upstream calls
architecting; the existing two are reviewers wearing the architect
label.

The `-roots` suffix is semantically opaque to new readers
(originally meant "structural roots" of pattern-purity, but reads
as ad-hoc to anyone outside the team).

## Scope

Pure rename. No functional changes.

- `agents/board-architect-roots.md` → `agents/board-pattern-reviewer.md`
- `agents/code-architect-roots.md` → `agents/code-pattern-reviewer.md`
- Internal self-references in those files updated (the personas
  describe themselves; rename their own self-name).
- Cross-references throughout the framework updated:
  - `agents/buddy/operational.md`
  - `agents/_protocols/spec-reviewer-protocol.md`
  - `agents/_protocols/code-reviewer-protocol.md`
  - `agents/_protocols/reviewer-base.md`
  - `skills/spec_board/SKILL.md` + `REFERENCE.md`
  - `skills/code_review_board/SKILL.md` + `REFERENCE.md`
  - `workflows/runbooks/build/workflow.yaml` + `WORKFLOW.md` + `REFERENCE.md`
  - `workflows/runbooks/fix/workflow.yaml` + `WORKFLOW.md`
  - `workflows/runbooks/review/workflow.yaml`
  - `framework/skill-map.md` + `boot-navigation.md`
  - `agents/navigation.md`
  - Any in-flight spec referencing the old names (notably spec 306
    if not yet locked).
- Search-and-verify pass: grep for residual `architect-roots` /
  `architect_roots` strings across the entire framework tree
  post-rename.

## Not yet (scope boundary)

- **No functional changes** — the personas continue to do exactly
  what they do today. Pattern-classes inherited, anti-rationalization
  prose, finding format — all unchanged.
- **No other persona renames** — only `architect-roots` → `pattern-
  reviewer`. Other personas (e.g. `board-adversary-2`) may have
  unclear suffixes too; those are separate audits.
- **No new pattern classes** — the rename does not introduce or
  remove any pattern-class taxonomy entries.
- **No refactor of architect-vs-reviewer role surface** — that is
  a deeper architecture question (does the framework keep both
  reviewer-personas AND a brief-architect-author?). Per spec 306
  §0 + §10.1, the answer is yes — three orthogonal phases. This
  task accepts that decision.
