# Task 300 — OSS-Translation des intentic-framework Repos (DE → EN)

## Problem

Das `intentic-framework` Repo ist heute zu ~98% in deutscher Sprache geschrieben
(Skills, Workflows, Agents, framework/, agents/buddy/, Spec-Templates, Discoveries).
Per User-Aussage 2026-05-04: "98% des Repos = OSS-readable, alles andere geloescht."

Konsequenz: solange dies nicht uebersetzt ist, ist die OSS-Methodology fuer
externe (englischsprachige) Beitragende nicht konsumierbar. Jede OSS-Push-
Aktivitaet (CONTRIBUTING.md, SECURITY.md, Mkdocs-Site, PyPI-Style-Release-
Notes, externe Issue-Diskussionen) ist davon blockiert.

Pyflue-Vergleich (`docs/research/2026-05-05-pyflue-vs-intentic.md`) zeigt das
explizit: pyflue hat OSS-Mature-Surface (CONTRIBUTING/SECURITY/CHANGELOG/
Mkdocs/PyPI/CI-Badges/dependabot) bei Library-Scope. Intentic hat unfertige
Surface bei breitem Process-Framework-Scope. Translation ist die untere
Schicht jeder weiteren Surface-Arbeit.

## Intent

Repo so umstellen, dass alle OSS-readable Files auf Englisch sind und der
projektkonsistente Fachbegriff-Wortschatz dokumentiert ist. **Zielzustand:**
ein englischsprachiger Beitragender kann `framework/`, `skills/`, `workflows/
runbooks/`, `agents/buddy/`, `intent.md`, `README.md` lesen und versteht die
Methodology ohne Uebersetzungs-Hilfe.

### 3-Phasen-Plan

#### Phase A — Glossar-Setup (30 Min)

**Output:** `docs/glossary-de-en.md` mit projekt-konsistenten Aequivalenten.

Beispiel-Eintraege (Pflicht-Konsistenz im gesamten Repo):
- "Spec-Engineering" → "Spec Engineering" (compound, kein hyphen-EN)
- "Reviewer-Persona" → "reviewer persona"
- "Pflicht" → "required" / "mandatory" (Kontext-abhaengig, im Glossar fixiert)
- "Pointer-Schema" → "pointer schema"
- "Source-Grounding" → "source grounding"
- "Werkbank" → "workbench"
- "Schicht" → "layer" (1+2 → "layer 1+2")
- "Drift" → bleibt "drift" (Englischer Lehnbegriff schon)
- "Discipline-as-Mechanism" → bleibt 1:1 (etablierter Begriff)
- "Source-of-Truth (SoT)" → bleibt 1:1 (etablierter Begriff)

Plus:
- **Test-File:** `agents/buddy/soul.md` als Pilot uebersetzen, Glossar-Drift
  sofort kontrollierbar.
- Reliable Estimate fuer Phase B + C nach Pilot.

#### Phase B — Critical Path (3-4h fokussiert)

**Scope:** ~30 OSS-readable Files, top-level Discovery-Pfade fuer externen Leser.

Konkrete Ziel-Files (Vorschlag, kann beim Start finalisiert werden):
- `intent.md`, `README.md`, `CLAUDE.md`, `AGENTS.md`
- `framework/` (alle ~20 Files: skill-anatomy, process-map, spec-engineering,
  agentic-design-principles, agent-patterns, intent-tree, milestone-execution,
  agent-autonomy, navigation, getting-started, etc.)
- `agents/buddy/{soul,operational,boot,context-rules}.md`
- `agents/_protocols/*.md` (reviewer-base, code-reviewer-protocol, etc.)
- `skills/_protocols/*.md` (evidence-pointer-schema, dispatch-template,
  consolidation-preservation, etc.)
- Top-level Skill `SKILL.md` der Tier-1 Skills (spec_board, code_review_board,
  council, frame, root_cause_fix, knowledge_processor, testing, scoping)
- `workflows/runbooks/*/WORKFLOW.md` (build, solve, fix, review, research,
  save, docs-rewrite, context_housekeeping)

#### Phase C — Full Pass (5-8h, optional)

**Scope:** alle 178 MD + 27 non-MD-Files konsistent.

- Restliche Skills (low-frequency Capability + Utility Skills)
- Discoveries (chronologisch durchgehen)
- archived Files NICHT (siehe "Noch NICHT")

- Pflicht-Skripte/YAML/Bash mit DE-Strings: nur Comments + User-facing Output-

## Dependency: blocked_by 299

Build 299 (Fabrication-Mitigation) hat den Source-Grounding-Layer mit
spezifischer Fachsprache eingefuehrt (Pointer-Schema, Reviewer-Protokoll-
Migration, schema_version, evidence_layout etc.). Diese Begriffe muessen im

Paralleles Laufen waere Mehr-Arbeit (300 wuerde Begriffe uebersetzen die 299
noch aendert).

## Constraints

### MUST

- **OSS-Tone:** keine BuddyAI-spezifische Forensik in Skills/Capability-Files
  (User-Lock L2 aus Build 299, gilt weiter).
- **Konsistenz:** Glossar ist authoritativ, Drift wird gefixt nicht akzeptiert.
- **Mechanik-Begriffe etabliert lassen:** "Source-of-Truth (SoT)",
  "Discipline-as-Mechanism", "Drift" — bleiben.

### MUST NOT

- KEIN Touch auf Code-Identifier (Funktions-Namen, Variable-Namen, Klassen-
  Namen). Nur Comments + User-facing Strings.
- KEIN Touch auf Companion-Soul + Memory Files (siehe "Noch NICHT").
- KEIN Touch auf personal-Context-Files (separate Repo).

### PREFER

- Englisch idiomatisch, nicht 1:1 woertlich. Lieber leicht umformulieren als
  steife Direkt-Uebersetzung.
- Quotes aus Spec/Code in EN, aber wenn Spec-File noch DE ist: Source-Quote
  bleibt DE bis Spec-File selbst uebersetzt ist.
- Phase B sequentiell statt parallel, Glossar-Drift sonst zu hoch.

## Noch NICHT (ausgeschlossen vom Scope)

- **Code-Identifier** (function names, variable names, class names, file names):
  bleiben EN (sind heute schon meist EN).
- **BuddyAI-Code-Repo** (`~/projects/BuddyAI/`): separate Repo, separater
  Translation-Task wenn ueberhaupt.
- **Personal-Context-Files** (`~/projects/personal/context/user/`): privat
  + DE-explicit, kein Touch.
- **Companion-Soul + Memory-Files** (`agents/companions/jc-denton/{soul,memory}.md`):
  privat, DE bleibt (Voice/Character-Konsistenz).
- **Session-spezifische Forensik-Discoveries**: nicht uebersetzen, sind Audit-Trail.
- **`skills/_archived/` + `framework/_archived/`**: keine Pflege, DE bleibt
  (sind Audit-Trail).
- **Commit-History**: keine Re-Writes, nur ab jetzt.
- **Inline-Quotes aus DE-Source-Files** wenn Source-File noch DE ist.
