# Task 308: Explore upstream CC repo for spec-authoring patterns

## Context

The framework's `spec_authoring` skill (`skills/spec_authoring/SKILL.md`)
is currently structured around interview-mode + 5 primitives + frame
sub-step integration. It was built before the upstream agentic-CLI
source was widely accessible.

The upstream Claude Code repo (cloned at `~/projects/ClaudeCode/`)
contains primitives that may be directly relevant to spec / requirement
authoring discipline:

- **Plan Mode** (`src/tools/EnterPlanModeTool/prompt.ts`) — full
  read-only session mode for "explore → understand patterns → design
  approach → present for approval". This is structurally a spec-
  authoring mode, not just plan-authoring.
- **Plan Agent** (`src/tools/AgentTool/built-in/planAgent.ts`) — the
  "Understand Requirements" phase (Phase 1) is canonical interview
  framing. The "Explore Thoroughly" phase is what the framework's
  source-grounding step approximates.
- **AskUserQuestion** patterns — how the upstream framework structures
  user interviewing during plan/spec authoring.
- **Memory hierarchy** (5-level cascade per
  `~/projects/ClaudeCode/docs/findings/005-memory.md`) — how authoring
  context flows into and out of the agent.
- **AgentTool brief-quality** + "Never delegate understanding" — how
  the framework's spec_authoring should brief sub-skills (frame,
  council).
- **Coordinator-mode quality-gate prompts** ("do not rubber-stamp",
  "understand findings before directing follow-up", "never hand off
  understanding") — anti-rationalization for the spec author role.

This task scopes the audit + mapping. Implementation of resulting
adoption items is a separate follow-up.

## Scope

Research-only. Concrete deliverables:

- **Inventory:** every CC source-tree primitive that touches spec /
  requirement / plan authoring. List file paths + one-line summary
  per primitive.
- **Mapping:** for each primitive, decide whether the framework's
  `spec_authoring` skill (a) already does this, (b) does it
  differently with a reason, (c) is missing this pattern.
- **Gap analysis:** the (c) entries become the candidate adoption
  list.
- **Recommendation:** which adoptions are high-value (would
  meaningfully improve spec quality), which are low-value (CC-
  specific conveniences not transferable). With rationale per item.
- **Follow-up tasks:** one task per high-value adoption, filed as
  separate task after this exploration.

## Workflow

- [ ] Step 1: Inventory pass — grep + read across `src/tools/`,
      `src/coordinator/`, `src/utils/`, plus the existing findings
      in `~/projects/ClaudeCode/docs/findings/`.
- [ ] Step 2: Read `skills/spec_authoring/SKILL.md` end-to-end with

- [ ] Step 3: Mapping table per primitive (a / b / c per above).
- [ ] Step 4: Gap analysis + recommendation per gap.
- [ ] Step 5: File follow-up tasks for high-value adoptions.
- [ ] Step 6: Output document summarizing findings as a finding
      file or skill-amendment proposal.
