# Task 303: Audit ‚Äî API-check + eval step coverage in workflows (DR-12)

## Context

Two source-grounding mechanisms exist (or did exist) in framework workflows:

1. **API-doc-check via `get_api_docs` skill** ‚Äî invoked in `build` and `fix`
   workflow steps before code is written against an external API/library.
   Purpose: prevent fabrication / version-drift on outdated training-data
   API knowledge. SoT: `skills/get_api_docs/SKILL.md`.
2. **Eval step** in `spec_authoring` (Phase 1, source grounding + interview)
   and `spec_amendment_verification` (post-MCA spec amendment check).
   Purpose: catch AC ‚Üî spec drift and amendment-introduced contradictions
   before they propagate downstream.

These steps are the framework's primary mechanical defence against the
fabrication failure-class (DR-12 source grounding,
`framework/agentic-design-principles.md`). If they have been weakened,
silently dropped, or renamed during recent restructuring (Task 299
fabrication-mitigation rewire, Task 300 EN translation, Task 301/302
fabrication-mitigation backlog passes), the framework loses its primary
defence ‚Äî and the loss is invisible until a board catches a fabrication
post-implementation.

Trigger: this question surfaced during the frame for the process-
improvements-from-task-469 work on 2026-05-10 ‚Äî while inventorying
existing primitives that overlap the brief-architect proposal, the user
asked whether the API-check and eval-step primitives are still present
AND sufficient.

## Scope

Audit-only. Two artefacts produced:

- A presence map: for each of `build`, `fix`, `spec_authoring`,
  `spec_amendment_verification` ‚Äî is the step in `workflow.yaml`, in
  `WORKFLOW.md`, in `REFERENCE.md`? Three columns, mark presence per file.
- A content judgment per step: does the step actually catch what it
  should? (API-check should grep for external-API imports + invoke
  `get_api_docs` before code is written, not after; eval should compare
  AC-set against spec sections; amendment-verify should compare amended
  spec against board-locked LDs.) Pass/weak/fail per step with a
  one-line rationale.

If a step is absent or weak: file a follow-up task to re-introduce or
strengthen it. The follow-up is out-of-scope of this audit ‚Äî this task
is verification only.

- [ ] Step 3: Content review ‚Äî for each present step, assess whether the

      still active (not retired), and `skills/spec_amendment_verification/SKILL.md`

- [ ] Step 5: Verdict per step ‚Äî present-and-sufficient / present-but-weak / absent.
      Total: 4 steps √ó 3 files = up to 12 cells; expect ~8 substantive verdicts.
- [ ] Step 6: If any cell is weak/absent: file a follow-up task with
      proposed strengthening. Do NOT redesign in this audit.
