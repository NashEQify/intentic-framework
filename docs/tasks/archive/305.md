# Task 305: Audit + refactor — MCA discipline

## Context

The upstream general-purpose Agent in the canonical agentic-CLI has a
~4-sentence system prompt: "Complete the task fully — don't gold-plate,
but don't leave it half-done", a brief strengths list, and three
guidelines about file creation. Tools allowlist is `*`. That is all.

This framework's `agents/main-code-agent.md` is substantially richer.
That richness is partly legitimate (we are spec-driven, the upstream is
not — we have a real contract surface to enforce), and partly cargo
cult (we tell the LLM HOW to think, not WHAT contract to honour). Trust
the LLM where the framework has no real differentiating reason to
prescribe; keep mechanical enforcement only where the contract
demands it.

## Three classes (discrimination)

### Class A — SDD contract (KEEP)

What the framework's spec-driven discipline genuinely requires. The
upstream baseline does not have these because the upstream is not
spec-driven. These are the legitimate differentiators:

- Intent alignment validation (intent_chain → STOP on deviation)
- Input validation per delegation path (Full / Standard / Backlog /
  Direct / Life — required fields)
- SPEC_VERIFICATION field on spec-touching changes
- Architecture escalation (>1 valid path → STOP, escalate)
- Stop-conditions (E1-E4 — spec contradiction, architecture
  decision, deps not satisfiable, tests still red after 2 fixes)
- Return summary with tester verdict
- Hard-limit "what MCA does NOT do" (no knowledge_processor, no
  context write, no history entry, no session-buffer write)

### Class B — Cognition prescription (TRIM)

What the LLM already does internally. Prescribing it explicitly
produces compliance theatre: the LLM performs the steps as ritual
rather than authentic reasoning. The upstream baseline has none of
these. Replace with behaviour rules ("recognize these rationalizations
when they arise") rather than process recipes.

- The 5-step Reasoning section (INTENT / PLAN / SIMULATE / FIRST
  PRINCIPLES / IMPACT). The LLM does these silently regardless;
  prescribing them turns reasoning into a checklist exercise. Replace
  with anti-rationalization heuristics in the upstream verification-agent
  style: "you say the code looks correct — reading is not verification.
  Run it." Behaviour rules, not process recipes.

### Class C — Operational hygiene (HOOK-IFY)

Mechanical conventions that belong in PreToolUse / PostToolUse hooks
or workflow-engine state, not in persona instructions. Hooks catch
mechanically and uniformly across agents; persona instructions are
unreliable because attention degrades under load.

- L0 static analysis (ruff + mypy gate before "done") — should be
  a PostToolUse hook on agent return, not an instruction MCA must
  remember to run.
- Observation masking (>50 lines / >2KB output redirected to file)
  — tool-use convention; should be a PreToolUse Bash hook or shell
  wrapper, not an instruction.
- Checkpoint writing (`workspace/.checkpoint.md` after every major
  step) — could be a PostToolUse hook on substantive tool calls,
  or a workflow-engine state-derived artefact.

## Workflow

- [ ] Step 1: Audit `agents/main-code-agent.md` line-by-line; tag
      every block with class A / B / C. Produce a delta diff.
- [ ] Step 2: Class B trim — rewrite the cognition section as
      anti-rationalization heuristics in the upstream verification-agent
      shape. ~5-10 lines, behaviour rules. Reference: upstream
      verification-agent system prompt's "Recognize your own
      rationalizations" block.
- [ ] Step 3: Class C hook design — for each hygiene item, decide
      hook kind (PreToolUse / PostToolUse) + scope. Spec the hook,
      file as separate sub-tasks if implementation is non-trivial.
- [ ] Step 4: Refactor MCA persona file to KEEP class A, replace
      class B, point class C at the new hooks.
- [ ] Step 5: Cross-check — does the trimmed MCA still satisfy all
      build-workflow contract requirements? Run a synthetic dispatch
      end-to-end.
- [ ] Step 6: Spec-board review (single-pass standard) on the
      refactored persona before lock.

## Not yet (scope boundary)

- **No change to other personas** — this audit is MCA-only. Other
  personas (board-architect-roots, code-chief, etc.) may have similar
  cargo cult, but each is a separate audit with its own contract
  surface.
- **No new SDD primitives** — the audit is about what to remove, not
  add. New SDD discipline (if the audit surfaces a gap) is filed as
  a separate task.
- **No retroactive change to past MCA behaviour** — existing tasks
  that ran under the rich MCA stay as-is; the refactor applies to
  future dispatches only.
- **No upstream-baseline cargo cult in reverse** — we keep the SDD
  contract (class A) even though upstream lacks it, because it
  legitimately differentiates this framework. The audit goal is
  trim where prescription is unjustified, not "make us look like
  upstream".
