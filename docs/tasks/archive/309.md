# Task 309: Audit framework workflows vs upstream coordinator-mode 4-phase

## Context

The upstream agentic-CLI's `coordinatorMode.ts` (Section 4) declares
a canonical 4-phase task workflow:

| Phase | Who | Purpose |
|---|---|---|
| Research | Workers (parallel) | Investigate codebase, find files, understand problem |
| Synthesis | Coordinator | Read findings, understand the problem, craft implementation specs |
| Implementation | Workers | Make targeted changes per spec, commit |
| Verification | Workers | Test changes work |

Workers parallelism is described as "your superpower"; concurrency
guidance: "Read-only tasks (research) — run in parallel freely".

The framework's build / fix / solve runbooks use a 5-phase model
(Specify / Prepare / Execute / Verify / Close) with different
boundaries:

- **Research-equivalent work** is currently distributed across:
  `spec_authoring` Phase 1 (source-grounding + interview),
  `frame` skill Step 4 (repo check) + Step 6 (SOTA conditional),
  optional standalone `research` workflow runbook.
- **Synthesis-equivalent work** is distributed across: spec writing
  in Specify, delegation-artefact authoring in Prepare.
- **Implementation** maps to Execute (MCA dispatch).
- **Verification** maps to Verify (code-review-board).

The framework does not have a workflow-built-in "parallel Worker
research before Specify" pattern. Research either fires standalone
(via `research` runbook) or happens inline by the orchestrator.

## Scope

Audit + gap-map + recommendation. Research-only. Concrete
deliverables:

- **Phase mapping:** for each of the 4 upstream coordinator-mode
  phases, list every framework artifact (workflow phase, skill,
  agent) that contributes to it. Show distribution; flag where
  multiple framework artifacts map to one upstream phase.
- **Gap identification:** where does the framework lack an equivalent
  to the upstream phase? Specifically: parallel-Worker research is
  the most prominent candidate gap.
- **Intentional-divergence cases:** where does the framework
  deliberately do it differently? Document why; check whether the
  rationale still holds.
- **Adoption-vs-divergence recommendation:** for each gap, propose
  one of: adopt-upstream / adapt-upstream / keep-divergence /
  defer-to-empirical-data.
- **Follow-up tasks:** if the recommendation is adopt or adapt, file
  a separate task per gap; this task scopes the audit only.
