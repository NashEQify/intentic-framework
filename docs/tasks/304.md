# Task 304: Multi-workflow Buddy-mode via TeamCreate-pattern adoption

## Context

The upstream agentic-CLI ecosystem has a primitive for persistent
multi-agent teams: a team config directory (with member registry +
agent-types), a shared task list at a known path, file-based mailbox
semantics for inter-agent messages, and async SendMessage delivery
between turns. Teammates "go idle" between turns, can claim unassigned
tasks from the shared list, and remain alive across sessions.

This framework currently runs one workflow at a time per Buddy session
(build OR fix OR solve OR review). Long-running workflows are tracked
via `workflow_engine.py` state files in `.workflow-state/<id>.json`,
but only one is active per process. There is no concurrent-team
infrastructure, no shared task list across teammates, and no
async-message-delivery primitive.

Adopting the team primitive would let a single Buddy session host
multiple concurrent workflows: e.g., two parallel builds on different
specs each with their own MCA dispatch + reviewer cycle, or a build +
fix running in parallel on independent code paths. Buddy would
context-switch between workflows by reading the shared task list and
delivered messages, while each teammate works asynchronously.

## Scope

Adoption + adaptation, not invention. Concrete deliverables:

- **Team primitive adapter** — map the upstream team config directory
  semantics onto the framework's existing artefact paths (`docs/tasks/`
  for the shared task list, `context/` for the team config, file-based
  mailbox at a path TBD).
- **workflow_engine multi-state** — current engine assumes one active
  state file per process. Add concurrent state semantics: lock
  granularity, state-file naming, cross-state reference rules.
- **Async-message delivery** — adapt the upstream auto-deliver-on-idle
  pattern to the framework's session model (one Buddy turn = one
  message-delivery point).
- **Buddy attention model** — Buddy soul / operational additions: how
  to context-switch between simultaneous workflows without losing
  thread continuity, when to surface inter-workflow conflicts,
  shutdown-request handling.
- **Task ownership across teammates** — extend `task_status_update`
  with an `owner` field analogous to upstream TaskUpdate; document
  reassignment rules.
- **Session-handoff multi-stream** — current handoff is one file
  per session; need either one-per-workflow or merged-with-sections
  semantics. Decide and spec.

## Workflow

- [ ] Step 1: Source-grounding — read upstream TeamCreate +
      TaskCreate/TaskUpdate/TaskList prompts and `coordinator/coordinatorMode.ts`
      end-to-end. Document the upstream contract surface.
- [ ] Step 2: Frame the adaptation: which upstream primitives map 1:1,
      which need framework-shaped equivalents, which are out-of-scope
      (e.g. tmux/iTerm pane management is irrelevant; KV-cache fork
      sharing is API-level, not application-level).
- [ ] Step 3: Spec authoring — `docs/specs/NNN-multi-workflow-session.md`
      with the 6 deliverables above. Spec-board review.
- [ ] Step 4: Implementation across affected modules (workflow_engine,
      session-handoff mechanism, Buddy soul/operational, task_status_update).
- [ ] Step 5: Verification — concurrent-build smoke test (two builds
      on synthetic specs, parallel teammate dispatch, no state corruption,
      clean shutdown).

## Not yet (scope boundary)

- **No KV-cache optimization** — that is a runtime-API concern, not
  application-level, and not transferable.
- **No terminal-pane / tmux integration** — the framework runs as an
  orchestrated process, not a terminal-multiplexed UI.
- **No `team_create` slash-command** — primitive lives at the
  workflow_engine + Buddy level, not as a user-visible CLI command.
- **No retroactive migration of in-flight workflows** — existing
  single-workflow sessions continue as-is; multi-workflow is opt-in
  for new sessions that explicitly enter the mode.
- **No persistent cross-host team state** — single-host first; multi-host
  team coordination is a separate concern.

## Dependencies

- The brief-architect / multi-architect spec (separately tracked) must
  land first — multi-workflow coordination assumes the framework already
  has a clean multi-agent dispatch pattern at the brief level. Without
  that, parallel builds would each re-invent dispatch discipline.
