### What upstream cc DOES NOT have

**No build- or task-completion gate.** cc has no build lifecycle ÔÇö no
"discrete work-unit with a defined end". Consolidation runs on
time/session cadence, not on work-completion. Coordinator's ┬º4 phase
table (`coordinatorMode.ts` lines 198-220: Research ÔåÆ Synthesis ÔåÆ
Implementation ÔåÆ Verification) ends at Verification ÔÇö there is no
Distill or Retro phase after it.

forge has the build concept (per spec-driven discipline) ÔåÆ the
build-end-boundary exists mechanically ÔåÆ a gate there is sensible,
but has no direct cc analog.

### What's liftable

**4-stage pattern from autoDream ÔÇö orient / gather / consolidate /
prune.** Clean retro/distill shape. Maps directly onto build_retro
inputs:

| autoDream stage | build_retro mapping |
|---|---|
| **orient** | Read brief + ACs + verdict + spec-drift diff + RETURN-SUMMARY (and optional friction-log) |
| **gather** | Collect findings: LD-outcomes, friction points, pattern emergence, stale-decision candidates |
| **consolidate** | Distill: ┬ºPatterns-Emerged + ┬ºFramework-Feed + ┬ºStale-Decisions + ┬ºFriction-Observations (1-pager structured) |
| **prune** | Discard noise: absorb-next-touch findings, duplicates, low-signal observations |

Plus the `target:` 6-value enum from spec 306 ┬º4.7 (chief-verdict
target-routing) for ┬ºFramework-Feed entries: `forge_task` /
`consumer_only` / `absorb_in_knowledge_process` / `discard`. Same
mechanism, different application.

### What's NOT liftable

- Background-fork mechanic (cc uses KV-cache-sharing via Anthropic
  API directly; forge runs as a CC consumer, transparent to the API)
- Time-cadence trigger (forge's trigger is build-end, not time)
- Per-turn memory-update (extractMemories runs cross-session, not
  per-build)

## Inputs ÔÇö what's needed at gate-time, what's optional

### Required inputs (already produced by earlier gates, no scaffolding needed)

- Brief (architect-authored at gate 5) ÔÇö or Buddy-inline on DIRECT path
- RETURN-SUMMARY from MCA (gate 7)
- code-review-board verdict (gate 8) ÔÇö chief-consolidated, with
  `remaining_findings:` + `target:` annotations
- spec-drift-check diff (gate 9) ÔÇö or empty if no drift
- Task ACs from `docs/tasks/<id>.{md,yaml}` (ground-truth checkpoint)

All exist at the gate-fire point as a natural consequence of the
build workflow's shape. No active scaffolding needed during earlier
gates.

### Optional input ÔÇö friction-log (proposed)

`docs/build/<slug>-observations.md` ÔÇö opportunistic append-only file
that anyone (Buddy, MCA, architect) can write to during the build to
capture transient observations: "this was friction", "this felt
surprising", "this is the 3rd time we've X". Format = build-468-learnings.md
shape, but lighter (no per-gate header obligation; just timestamped
notes).

retro **reads if present, ignores if absent**. No discipline obligation
to populate it ÔÇö opportunistic only. Build-468-learnings.md is the
existence-proof: when the friction-log gets populated, the value is
high; when not, the artifact-only-retro still works.

This is the lightweight echo of the retired authority-log: structured
input file for the retro step. Difference: opportunistic (not mandated),
unstructured (free-form notes), retro-consumed (not workflow-validated).

### NOT needed

- Authority-file (retired with audit, deliberately not revived) ÔÇö its
  decision-record role is filled by brief + RETURN-SUMMARY + ADR
- Per-gate "what surprised you" prompt ÔÇö adds discipline tax for
  marginal gain; opportunistic friction-log covers the same ground

## close-bookkeeping consolidation

### Current shape (build/WORKFLOW.md gate 10)

Three skip-eligible sub-checks:
- ADR-discipline triple (hard-to-reverse + surprising-without-context +
  result-of-real-trade-off ÔåÆ write ADR)
- risk follow-up (file ONE follow-up task per non-empty
  `remaining_findings:` block in verdict files)
- knowledge-process (`knowledge_processor mode=process`)

Each sub-check independently reads the verdict / source artifacts and
does its own micro-extraction.

### Overlap analysis

| close-bookkeeping piece | retro overlap | merge / keep |
|---|---|---|
| ADR-discipline triple | retro ┬ºStale-Decisions + ┬ºPatterns-Emerged feed ADR-decisions ("which decisions met the triple, are surprising, or shifted") | merge ÔÇö retro distills, ADR triple-check fires on retro output |
| risk-followup-routing (target-routing chain) | retro ┬ºFramework-Feed and ┬ºFriction-Observations are routing candidates with the same `target:` enum mechanism | merge ÔÇö retro produces the typed entries that risk-followup-routing routes |
| knowledge-process | retro ┬ºPatterns-Emerged is exactly KP's input (distilled patterns worth persisting) | merge ÔÇö KP runs ON retro output, not from raw artifacts |

### Restructured close phase (proposed)

Single phase with two halves:

1. **distill** = build_retro (orient ÔåÆ gather ÔåÆ consolidate ÔåÆ prune)
   produces structured retro 1-pager.
2. **emit** = ADR-write-if-needed + risk-followup-route +
   knowledge-process ÔÇö all consume retro output.

Pseudocode-shape for the emit phase:

```
retro_output = build_retro.run(inputs)

# emit
for decision in retro_output.stale_decisions + retro_output.patterns_emerged:
    if adr_triple(decision):  # hard-to-reverse + surprising + trade-off
        write_adr(decision)

for entry in retro_output.framework_feed:
    risk_followup_routing.dispatch(entry)  # uses same target: enum

if retro_output.patterns_emerged:
    knowledge_process(input=retro_output.patterns_emerged)
```

Cleaner than three independent micro-extractions from the same
artifacts. Produces a single retro-1-pager that's the audit trail of
"what this build taught us", which the emit-step consumes.

### Trigger

Conditional, not always. Default: skip-eligible. Fires on:

- FULL build path (always)
- STANDARD path with `--require-retro` flag set on the workflow start
- New-pattern signal from architect's brief (LD class
  `new-class-{name}`)
- User-explicit override

Sub-threshold (DIRECT, STANDARD-routine, AUTHORITY-ONLY): skip with
one-liner rationale, like other conditional gates.

### Output shape

1-pager structured markdown at `docs/build/<slug>-retro.md`:

```markdown
# Retro: <task-id> ÔÇö <task-title>

**Build path:** STANDARD | FULL
**Inputs read:** brief, RETURN-SUMMARY, verdict, spec-drift-diff, AC-list
  [, observations-log if present]

## ┬ºPatterns-Emerged

- Pattern: <one-line>
  Evidence: <where it surfaced ÔÇö gate / artifact / line>
  Codify-target: <which framework file would hold this>

## ┬ºFramework-Feed

For each candidate, target-routed per the 6-value enum:

- target: forge_task ÔÇö <one-line, what new task in forge-repo>
- target: consumer_only ÔÇö <one-line, applies to consumer not forge>
- target: absorb_in_knowledge_process ÔÇö <one-line, KP-eligible>
- target: discard ÔÇö <one-line, with reason>

## ┬ºStale-Decisions

- Decision: <quoted from brief / earlier gate>
  Why-stale: <what later gate / discovery invalidated it>
  Action: <retain-as-ADR / supersede / no-action>

## ┬ºFriction-Observations

(only if observations-log was present, or if patterns surfaced
qualitative friction)

- <one-line observation, source-tagged>
```
