# Task 313: Codex Plugin Export

## Capability-portability analysis (input for spec phase)

This task created a first-pass capability audit. The corrected matrix —
after pushback that pre-commit ≠ PreToolUse and that setup-mechanisms
≠ capability-gaps — is captured here so the spec phase doesn't repeat
the work.

**Tier-1 (standards-portable across runtimes)**

| Capability | Mechanism |
|---|---|
| Skill format | `SKILL.md` + frontmatter (CC + Codex parse identically) |
| Workflow runbooks | markdown + YAML, runtime-independent |
| `workflow_engine.py` | Python CLI + JSON state files, runnable anywhere |
| Git pre-commit | 12 checks, harness-agnostic |
| Post-commit dashboard | git hook |
| Sub-agent dispatch | each runtime brings its own (CC `Task`, OC `SendMessage`, Codex subagents) — *not* a framework concern |

**Tier-2 (Claude-Code-coupled, no cross-runtime standard exists)**

| Capability | Why CC-only |
|---|---|
| `path-whitelist-guard`, `frozen-zone-guard`, `state-write-block`, `engine-bypass-block` (PreToolUse) | Prevention *before* the tool-call writes a file. No PreToolUse-class standard across runtimes. Pre-commit catches *after* the LLM polluted its context — **not a substitute** (rollback ≠ prevention). |
| `mca-return-stop-condition`, `board-output-check` (PostToolUse:Task) | Tool-output inspection in the agent loop |
| `workflow-reminder` (UserPromptSubmit) | Per-turn injection — Codex/OC use boot-time skill-loading, not per-turn |
| `delegation-prompt-quality`, `plan-adversary-reminder` | Quality enforcement on dispatch — coupled to the harness's dispatch tool |

**Per-harness setup (not a portability concern, just docs)**

| Setup item | Per-harness form |
|---|---|
| Framework path injection | CC: `--add-dir`; OC: `OPENCODE_CONFIG_DIR`; Codex: plugin install |
| Persona/skill discovery | CC: `~/.claude/agents` symlink; OC + Codex: bundled in plugin/config |
| Env vars | shell rc (per-harness convention) |

**Key correction (load-bearing for spec phase):**

> Pre-commit hooks fire at *commit time*, after the LLM already wrote
> the change to its context. PreToolUse hooks fire *before* the write
> happens. These are categorically different — prevention vs. rollback-
> protection. Earlier "PreToolUse → pre-commit, 80% portable" framing
> conflated them. Honest answer: PreToolUse is **0% portable** as
> standards-form. The prevention layer is CC-coupled.

**Implication for Codex plugin export (this task):**

A Codex plugin built from forge can ship:

- ✓ Skills (SKILL.md format compatible)
- ✓ Workflow definitions (markdown runbooks)
- ✓ `workflow_engine.py` as MCP server (state + step pointer)
- ✓ Post-hoc validation (anything triggered after a step completes)
- ✓ Git pre-commit hooks (in repos using the plugin)

A Codex plugin **cannot** ship:

- ✗ Mechanical prevention (PreToolUse-class blocking) — Codex doesn't
  have an equivalent hook surface as of May 2026
- ✗ Per-turn workflow-reminder — Codex auto-loads skills by description
  match, not per-turn injection

**Spec phase must explicitly state this asymmetry** in the plugin's
README and in `intent.md` adapter table — overpromising prevention
on Codex would mislead users.

This matrix supersedes any earlier capability claims; treat it as
the input for spec-write step.

## Background

OpenAI Codex CLI ships a manifest-driven plugin format (v0.117.0+,
March 2026). A Codex plugin bundles three things:

1. **Skills** — `skills/<name>/SKILL.md` with frontmatter (`name`,
   `description`). **Identical format to our skills.**
2. **MCP Servers** — `.mcp.json` mapping `mcp__<server>__<tool>`.
3. **App Connectors** — `.app.json` for GitHub/Slack/Drive integrations.

Pivotal Codex spec quote:
> *"Skills are the authoring format; plugins are the distribution format.
> A plugin is not a replacement for an MCP server."*

We already produce in the authoring format. What we don't have is the
distribution wrapper — neither manifest nor MCP-wrap of our state
engine.

## Why this matters

Tool-neutrality (intent.md pillar) gains a concrete distribution channel.
Currently we adapt via `--add-dir $FRAMEWORK_DIR` (CC),
`OPENCODE_CONFIG_DIR` (OC), Cursor rules. Each adapter is hand-written
glue. A Codex plugin is a *standard package format* — install via
`/plugins`, browseable in the marketplace, no per-consumer adapter code.

For small teams (intent.md secondary audience), a published Codex plugin
lowers the adoption floor enormously.

## Sub-components (not sub-tasks — single coherent build)

1. **plugin.json manifest generator** — script that reads the active
   skill inventory + workflow runbooks and produces a valid
   `.codex-plugin/plugin.json` (`name`, `version`, `description`,
   `skills: ./skills/`, `mcpServers: ./.mcp.json`, `apps: ./.app.json`,
   `interface: {...}`). Required fields per
   developers.openai.com/codex/plugins.

2. **workflow_engine.py as MCP server** — the core architectural piece.
   Wrap the engine's CLI surface (`--next`, `--complete`, `--pause`,
   `--resume`, `--validate`, `--boot-context`) as MCP tools. Without this,
   a Codex plugin export ships only skills (text), not the discipline
   mechanism. **This is the substantial engineering chunk.** Cross-cutting
   benefit: same MCP-server makes the engine portable to *any* MCP-aware
   harness, not just Codex.

3. **Skill frontmatter audit** — scan `skills/*/SKILL.md` for fields
   that are CC-specific (`disable-model-invocation`, `invocation`, `uses`,
   `status`). Decide per field: keep as Codex-compatible, drop, or
   document as CC-only. Codex-spec frontmatter is minimal: `name` +
   `description`. Anything else may need tolerant parsing or namespacing.

4. **Hook audit** — `path-whitelist-guard`, `frozen-zone-guard`,
   `state-write-block`, `engine-bypass-block`, `delegation-prompt-quality`,
   `mca-return-stop-condition`, `board-output-check`, `workflow-reminder`,
   `plan-adversary-reminder` are all CC `PreToolUse`/`PostToolUse`/
   `UserPromptSubmit` hooks. For Codex: which of these can be MCP-tools
   (callable validate-X functions), which can be pre-commit-only
   (harness-agnostic), which stay CC-exclusive (drift in OC/Codex)?

5. **marketplace.json** — emit a `.agents/plugins/marketplace.json`
   (repo-scoped) entry pointing at the plugin path with
   `"source": "local"`, `policy.installation: "AVAILABLE"`.

6. **`orchestrators/codex/` adapter** — analog to `orchestrators/opencode/`
   and `orchestrators/claude-code/`. Holds the manifest generator,
   MCP-server wrapper entry-point, and any Codex-CLI-specific config.

## Dependency: blocked_by N

`blocked_by: []` (no hard blocker). Soft sequence preference: 298
(OpenCode adapter) first. Reasoning:

- 298 establishes the harness-agnostic-primitive pattern
  (`orchestrators/<runtime>/` adapter shape, hook-translator pattern,
  what's-translatable-vs-CC-only learnings).
- Codex distribution is *structurally different* (plugin-bundle, not
  hook-translation), so it doesn't strictly require 298 complete.
- Could run in parallel; preferred sequence is 298 first to avoid
  rework if 298 surfaces a "engine-as-MCP-server" approach that Codex
  also wants.
- The MCP-wrap of `workflow_engine.py` is the cross-cutting piece —
  spec'ing it as a runtime-agnostic primitive (sub-component 2 above)
  could *feed back* into 298. That's the architectural argument for
  not deferring 313 indefinitely.

## Acceptance Criteria (draft — sharpened on spec-write)

- AC-1: A reference Codex plugin can be assembled from this repo using
  `orchestrators/codex/build.sh` (or equivalent), producing a valid
  `.codex-plugin/` directory that passes Codex's plugin-validate.
- AC-2: `workflow_engine.py` can be invoked as an MCP server (e.g.
  `python3 scripts/workflow_engine.py --mcp`), exposing a documented
  set of MCP tools (`workflow_next`, `workflow_complete`, …).
- AC-3: A test consumer (Codex CLI) can install the plugin from a local
  marketplace.json entry and invoke at least one workflow end-to-end
  (e.g. `solve` light-path) using the MCP-wrapped engine.
- AC-4: Hook-audit document lists every active hook with a routing
  decision (MCP-tool / pre-commit-script / CC-only).
- AC-5: Skill-frontmatter-audit document lists every framework-specific
  field with a Codex-compatibility decision.

## Not yet (scope boundary)

- **Not a 1.0-blocker.** Roadmap-Item. Discipline-tightening + skill-inventory
  expansion (tool-use, multi-agent/teamwork) take priority.
- **Not a marketplace push.** Local marketplace.json only; submitting to
  Codex's curated/built-in marketplace is a separate downstream task once
  the local export is proven.
- **Not replacing existing CC adapter.** Codex adapter is additive; CC
  remains the primary runtime for now.
- **Not changing skill format substantially.** Skills already
  SKILL.md+frontmatter; audit may surface minor adjustments, not a
  rewrite.
- **Not an OpenCode equivalent.** OC has a different lifecycle model
  (Task 298). 313 is Codex-specific. Cross-cutting MCP-wrap of the
  engine *may* benefit OC, but that's discovered, not promised.
- **No App Connectors in v1.** `.app.json` is optional in the Codex
  spec; framework has no GitHub/Slack/Drive integrations to expose yet.
