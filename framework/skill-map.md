## Skill taxonomy

| Type | What it is | How it is loaded |
|-----|-----------|-------------------|
| **Workflow** | end-to-end flow with phases | Buddy follows the runbook |
| **Capability** | one callable capability | Buddy dispatches agent(s) with the skill |
| **Protocol** | rule-set loaded by others | Agent loads it in addition to main skill |
| **Utility** | single function | Buddy or agent calls directly |

### Architecture model (isomorphic to "6 Skills")

Workflows are **entry points** — routing trees that compose capabilities.
Capabilities are **work units** — the actual methodology.
Protocols are **rules** loaded into capabilities.
Utilities are **simple capabilities** without multi-phase complexity.

This model is isomorphic to OpenAI's "6 Skills" pattern: their skills =
our workflows (entry points), their internal skill code = our
capabilities. Few entry points, modular building blocks behind them.

New capabilities -> integrate into existing capabilities/workflows
instead of creating new skills. Consolidation over proliferation
(DR-7 cognitive overhead limit).

**Skill guardrails (required per skill):** every skill has boundaries
(routing protection, always required) and — for workflow/capability —
anti-patterns (execution protection). Details:
`skills/_protocols/skill-guardrails.md`. Enforcement via spec_board review.

## Cross-workflow handoffs

Workflows can invoke other workflows as sub-workflows:

| Parent | Sub-workflow | Trigger | Return |
|--------|-------------|---------|--------|
| Solve (Frame) | Research | SOTA trigger in frame step 6 | findings -> frame report |
| Solve (Refine) | Council | >1 path, hard to reverse | decision -> sharpened approach |
| Solve (Execute) | Build / docs-rewrite / fix | artifact type determines handoff target | next workflow starts |
| Build (Specify) | Research | step 9: pattern check after board | findings -> spec context |
| Build (Verify) | Fix | code-review FAIL -> fix required | fix committed -> re-review |
| Build (Specify) | Review | board run is review sub-flow | board_result -> gate |
| Fix (Specify) | Research | root cause unclear, broader analysis needed | hypothesis -> fix scope |

**Mechanics:** parent pauses -> sub-workflow runs (own phases) -> return
artifact (findings, fix, board_result) goes back to parent -> parent continues.
No nesting deeper than 2 levels. Sub-workflow has its own iteration bounds.

## Spec-related skills lifecycle (addendum)

The 4 spec skills form one lifecycle with clear transitions:

```
spec_authoring              (NEW spec or new section, interview-based)
     ?
retroactive_spec_update     (EXISTING spec, catch up to as-is code state)
     ?
spec_amendment_verification (cross-spec consistency check)
     ?
spec_board                  (rebuild-readiness quality review)
```

**Intent boundaries** (why 4 skills instead of 1):

| Skill | Intent | Input | Output |
|---|---|---|---|
| spec_authoring | create new spec/section (interview) | user intent, solution space | new spec/section with artifact checklist (incl. implementation-surface estimate) |
| retroactive_spec_update | align existing specs to as-is code state | target specs + source code | updated specs along 5 primitives |
| spec_amendment_verification | cross-spec consistency after amendment | changed specs | PASS / ISSUES_FOUND |
| spec_board | rebuild-readiness quality check | spec + 5 dimensions | verdict PASS / NEEDS-WORK |

All four share the guiding motif: spec defines the product, incidents
are fixed in the spec not in code. See `framework/spec-engineering.md`.

**Authoring chain (where each artifact lives):**
`spec_authoring` (greenfield spec, Buddy-direct interview) →
`agents/brief-architect.md` (per-task brief at delegation time;
default `mode=brief`) → `agents/main-code-agent.md` (implementation).
spec_authoring is the spec-time boundary; brief authoring at
delegation time is brief-architect's job, not spec_authoring's.

**Architect modes (per spec 306 §14):** `brief-architect` is one
persona with three modes. Default `mode=brief` for MCA delegation
briefs (above). `mode=spec_amendment` is dispatched from
`spec_authoring` "## Amendments to existing specs" when the
substantial-amendment threshold fires (cross-ref cascade ≥3 OR
cross-spec coupling OR class-rename / mechanism-shift /
contract-retraction). `mode=retro_spec_update` is the default Phase
2 path of `retroactive_spec_update` — the architect walks each
spec section against the source code and returns findings inline.
In all three modes the architect is read-only (no Write target in
`spec_amendment` and `retro_spec_update`; brief mode keeps the
single brief-output Write exception); the orchestrator integrates
the prose and does the final write.
